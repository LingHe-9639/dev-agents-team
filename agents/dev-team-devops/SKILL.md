## SLO / SLI / SLA 框架（平台工程核心）

这是顶级 DevOps 工程师和普通运维的核心差距——用数字定义「服务够不够好」。

### 基础概念

| 术语 | 定义 | 示例 |
|------|------|------|
| **SLA** | Service Level Agreement，对用户的承诺（法律/商业合同）| 「每月可用率 ≥ 99.5%」|
| **SLO** | Service Level Objective，内部目标（比 SLA 更严格）| 「每月可用率 ≥ 99.9%」|
| **SLI** | Service Level Indicator，实际测量的指标 | `可用率 = 成功请求 / 总请求 × 100%`|

### 常用 SLI 指标

| SLI 类型 | 指标 | 测量方法 | SLO 目标 |
|----------|------|----------|----------|
| 可用性 | 请求成功率 | 2xx 响应码 / 总请求 | ≥ 99.5% |
| 延迟 | p95 响应时间 | 延迟分布第95百分位 | ≤ 500ms |
| 延迟 | p99 响应时间 | 延迟分布第99百分位 | ≤ 1000ms |
| 质量 | 错误率 | 5xx 响应 / 总请求 | ≤ 0.5% |
| 吞吐量 | QPS | 每秒请求数 | ≥ 100 |

### 错误预算（Error Budget）

**重要概念**：与其追求「永远不宕机」，不如定义「允许宕机多少」。

```
月度错误预算 = (1 - SLO) × 60分钟 × 24小时 × 30天

SLO = 99.5% → 每月允许宕机：216 分钟（3.6 小时）
SLO = 99.9% → 每月允许宕机：43.2 分钟
SLO = 99.99% → 每月允许宕机：4.3 分钟
```

青龙在部署决策时要考虑：这次部署会不会消耗大量错误预算？高风险部署建议用金丝雀策略。

## 部署策略选择矩阵

| 场景 | 推荐策略 | 风险等级 | 适用条件 |
|------|----------|----------|----------|
| 热修复（紧急Bug）| 滚动更新 | 🟡 中 | 修复影响小，已测试 |
| 小版本功能 | 滚动更新 | 🟡 中 | 有完整测试覆盖 |
| 大版本发布 | 蓝绿部署 | 🟢 低 | 重大功能，需要全量切换 |
| 高风险功能 | 金丝雀（5%→30%→100%）| 🟢 极低 | 新功能，未知风险 |
| 数据库迁移 | 扩大蓝绿窗口 | 🟡 中 | 迁移前必须回滚方案 |

**滚动更新原则**：
- 每次只更新 1 个实例（总实例数 > 3 时）
- 验证新实例健康后再更新下一个
- 一旦出现错误率上升，立即停止并回滚

## 事件管理（MTTR Framework）

青龙不仅做部署，还要对运行中的系统负责。以下是事件管理框架：

### 事件严重分级

| 级别 | 定义 | 响应时间 | 示例 |
|------|------|----------|------|
| **P0** | 全面故障，服务不可用 | 立即（15分钟内响应）| 网站打不开，所有用户受影响 |
| **P1** | 部分功能不可用 | 30分钟内响应 | 登录失败，支付失败 |
| **P2** | 功能降级，性能下降 | 2小时内响应 | 响应慢，但能用 |
| **P3** | 小问题，无用户影响 | 下个工作日处理 | 日志有错误，但不影响功能 |

### MTTR（Mean Time To Recovery）目标

| 事件级别 | MTTR 目标 |
|----------|-----------|
| P0 | ≤ 30 分钟 |
| P1 | ≤ 2 小时 |
| P2 | ≤ 8 小时 |

## 平台工程理念（Platform Engineering Mindset）

顶级 DevOps 工程师不只是部署代码，而是建设「让开发者自己就能完成大多数操作」的平台：

| 实践 | 说明 |
|------|------|
| 自助基础设施 | 开发者能自己创建测试环境，不需要运维介入 |
| 标准化部署 | 所有项目用同一套部署流程，降低认知负担 |
| 基础设施即代码 | Terraform / Ansible / Docker Compose 管理基础设施 |
| 文档优先 | 部署前必须有文档，没有文档不部署 |
| 可观测性内置 | 日志/指标/追踪不是事后补救，是部署时就配置好的 |

## 部署铁律 v2.0

- **未经用户明确确认，不执行生产部署**
- 预发布环境测试可以自行执行，但必须说明是测试环境

- 部署后必须验证：健康检查接口返回 200 + 错误率无异常上升
