### 2. 性能预算（必须给出具体数字）

这是顶级架构师和普通架构师的核心差距——不只是说「要快」，而是说「快到多少」。

| 指标 | 目标值 | 测量方法 | 超标应对 |
|------|--------|----------|----------|
| API p95 响应时间 | ≤ 200ms | JMeter / pytest-benchmark | 加缓存 / 异步处理 |
| 首屏加载（桌面） | ≤ 2.5s | Lighthouse | 代码分割 / CDN |
| 数据库单次查询 | ≤ 50ms | EXPLAIN ANALYZE | 加索引 / 优化 SQL |
| 最大并发连接 | ≥ 100 | 压测工具 | 水平扩展方案 |
| 可用率 | ≥ 99.5% | 全年允许宕机 4.38h | 主备方案 |

### 3. 模块划分（C4 模型视角）

用三个层次描述模块结构：

**系统上下文**（Context）：这个系统和外部世界的交互关系
```
用户 → [本系统] → 微信API / 数据库 / 文件存储
```

**容器划分**（Container）：主要的技术容器（服务/应用/数据库）
```
[Web服务: FastAPI] ←→ [数据库: SQLite]
[前端: HTML/JS] → [Web服务: FastAPI]
```

**组件详情**（Component）：每个容器内的关键组件

| 组件 | 职责 | 不做 | 依赖 |
|------|------|------|------|
| auth_handler | 用户认证和会话管理 | 不做业务逻辑 | 无 |
| article_service | 文章的 CRUD 操作 | 不做搜索优化 | auth_handler, db |
| notification | 消息推送 | 不做发送失败重试 | article_service |

### 5. 演进式架构策略（Evolutionary Architecture）

顶级架构不是一步到位，而是设计「最容易下一步怎么改」的架构。

**Strangler Fig 模式**（如果涉及替换旧系统）：
- 新功能如何逐步接管旧功能（按模块、按 API 逐个替换）
- 切换条件是什么（灰度比例 / 功能开关）

**扩展方向**（提前说明本次不支持但架构预留了空间）：
- 本次不支持多租户，但数据库设计考虑了租户隔离字段
- 本次单节点部署，但 API 设计无状态，支持未来水平扩展

### 6. 安全架构（内置在架构里，不事后打补丁）

| 层次 | 措施 | 说明 |
|------|------|------|
| 边界 | HTTPS + HSTS | 传输加密 |
| 认证 | Bearer Token / JWT | 有过期时间 |
| 授权 | RBAC（角色权限控制） | 每个接口检查权限 |
| 输入 | 参数校验 + 白名单 | Pydantic schema |
| 查询 | 参数化查询 | 防止 SQL 注入 |
| 输出 | 通用错误信息 | 不泄露内部细节 |
| 密钥 | 环境变量注入 | 不在代码中存凭证 |

## 架构设计原则 v2.0

1. **不炫技**：选型优先考虑「团队能不能维护 3 年」，不是「哪个最前沿」
2. **权衡公开**：每个选择说明「放弃了什么」，不只说「得到了什么」
3. **性能预算先行**：在写代码之前就知道「快到哪里算及格」
4. **安全内置**：安全不是事后检查，是架构设计的一部分
5. **演进预留**：今天的设计要为明天留扩展口，但不要过度设计
6. **简单性优先**：用 3 个组件能做的事，不要用 5 个
